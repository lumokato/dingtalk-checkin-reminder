# 打卡提醒运行流程

## 目标

应用只做提醒和拉起钉钉，不伪造打卡。真实打卡由钉钉极速打卡完成。

## 上班流程

1. 用户在规则页保存地点、半径、上班时间、允许迟到时间。
2. 开启守护后，应用安排上班窗口闹钟。
3. 默认上班窗口为 `上班时间 - 30分钟` 到 `上班时间 + 允许迟到分钟`。
4. 到窗口后，前台服务进入自适应检测：
   - 保持前台通知。
   - 请求定位更新。
   - 每次定位/解锁/定时 tick 都重新判断。
   - 解锁/亮屏广播只是触发器之一，不能作为唯一依据。服务运行中还要在 tick 里读取 `PowerManager + KeyguardManager`，确认手机是否真的已解锁。
   - 上班窗口内未完成时，靠近目标范围后使用 5 秒轻量 tick 轮询解锁状态；定位请求仍按距离和运动趋势控制，不因为 tick 变成全局 GPS 高频。

## 上班通勤状态定义

应用不读取路线，也不猜用户目的地，只按下面条件判断“可能正在去上班”：

1. 今天是中国工作日或调休上班日。
2. 当前处于上班检测窗口：`上班时间 - 30分钟` 到 `上班时间 + 允许迟到分钟`。默认是 `08:00-09:00`。
3. 今天还没有记录上班完成。
4. 位置满足以下任一条件：
   - 亮屏/解锁：强制 `15秒` 快速定位探测，且 `60秒` 内最多触发一次。
   - 已在打卡半径内。
   - 在打卡半径外 `0-50米`。
   - 在打卡半径外 `50-500米`，并且和上一次距离相比向目标减少 `>5米`。这是“通勤中”的启动条件，不是持续条件。

如果范围外 `50-500米` 但没有明显接近，也就是和上一次距离相比向目标减少 `<=5米`，定位请求保持低频 `60秒 / 30米`。但为了不依赖解锁广播，服务 tick 仍每 `5秒` 轻量读取一次解锁状态；一旦发现已解锁，就进入 `15秒` 快速定位探测，用于刷新旧位置。
一旦在上班窗口内触发“通勤中”，直到上班完成或上班窗口结束，定位保持 `15秒 / 5米`，不再回退到 `60秒`。
如果范围外超过 `500米`，即使在上班窗口内也只保持远距离低频。

## 自适应检测参数

目标半径默认为 300 米。下面的“范围外距离”指 `当前位置到目标点距离 - 目标半径`。

| 状态 | 条件 | 评估频率 | 定位请求 |
| --- | --- | --- | --- |
| 已在范围内 | 距离 <= 半径；触发后锁存到上班完成或窗口结束 | 每 5 秒轮询解锁；收到或检测到解锁立即触发 | 连续定位暂停 |
| 边界附近 | 范围外 0-50 米 | 每 5 秒 | 5 秒 / 0 米 |
| 通勤中 | 范围外 50-500 米，且曾经和上一次距离相比向目标减少 >5 米；触发后锁存到上班完成或窗口结束 | 每 5 秒轻量轮询解锁 | 15 秒 / 5 米 |
| 未明显接近 | 范围外 50-500 米，且和上一次距离相比向目标减少 <=5 米 | 每 5 秒轻量轮询解锁 | 60 秒 / 30 米 |
| 远距离 | 范围外超过 500 米 | 每 60 秒 | 60 秒 / 30 米 |
| 刚亮屏/解锁定位探测 | 工作日、上班打卡窗口、未完成 | 15 秒内每 5 秒，60 秒冷却 | 5 秒 / 0 米 |
| 刚解锁边界探测 | 工作日、打卡窗口、未完成，且尚未锁存范围内、位置在范围外 0-50 米 | 10 秒内每 5 秒 | 5 秒 / 0 米 |

这些数值是重要行为参数，后续如果调优必须同步修改代码和本文档。
5. 进入范围且手机已解锁时：
   - 立即拉起钉钉。
   - 拉起后如果已授权使用情况访问，必须验证钉钉是否真的进入前台。
   - 如果钉钉没有进入前台，立即发强提醒，并在短时间内重试；不能等普通重复提醒间隔。
   - 同时发一条强提醒通知，通知里保留“打开钉钉”和“已上班打卡”动作。
   - 进入范围状态会锁存；触发拉起后停止连续定位，避免在公司范围内继续 5 秒高精度定位。
   - 不能用过期定位触发进入范围；定位年龄超过 120 秒时必须先刷新定位。
6. 如果钉钉成功通知能被识别，应用自动记录上班完成。
7. 如果开启“钉钉进入前台后自动记录”，必须先授权使用情况访问，并且系统 Usage Events 能确认钉钉确实进入前台；否则不能自动记录成功。
8. 如果自动识别失败且没有通过前台验证，用户必须通过通知确认或后续手动确认入口记录完成。
9. 如果上班窗口结束时仍未确认成功，应用记录“上班未确认”，清掉上班提醒，并继续安排当天正常下班提醒。这个状态不能阻塞下班流程。

## 下班流程

1. 下班基础时间为规则页的下班时间。
2. 如果已记录上班，并且上班晚于正常上班时间，则下班提醒顺延相同迟到时间。
3. 如果未记录上班，或上班窗口结束后仍是“上班未确认”，则按正常下班时间提醒。
4. 到下班窗口后：
   - 默认不要求仍在范围内。
   - 如果规则开启“下班也要求在范围内”，则先判断范围。
   - 默认不循环强提醒；手机解锁后立即拉起钉钉。
   - 成功尝试拉起钉钉后只发一次可确认通知，避免持续提醒。
5. 钉钉成功通知能识别则自动记录下班完成；如果开启“钉钉进入前台后自动记录”，必须确认钉钉确实进入前台后才能兜底记录；否则走通知手动确认。

## 首页应急入口

首页右上角胶囊复用为最短路径入口：

- 守护未开启：点击开启守护。
- 守护已开启：点击直接打开钉钉。
- 配置缺失时：点击进入规则页。

## 失败兜底

应用必须尽量暴露失败状态：

- 等待定位时，通知说明正在等待定位。
- 不在范围内时，前台通知显示当前距离。
- 系统限制后台拉起钉钉时，强提醒通知保留“打开钉钉”动作。
- 拉起钉钉后如果没有识别到成功通知，也没有验证钉钉确实进入前台，不自动标记完成，避免误记。
- 已授权前台验证时，拉起钉钉后必须记录钉钉是否真的进入前台；未进入前台要立刻提醒并短时间重试。

## 当前代码要求

- 上班/下班是否等待定位、等待新定位、等待解锁、打开钉钉，必须优先通过纯决策层判断，再由服务执行通知、定位、启动 Activity 等副作用。
- 上班窗口内检测频率必须按距离/运动趋势自适应，具体数字见“自适应检测参数”。
- 定位请求不能全局固定高频；远距离低频，近距离高频，边界附近和符合通勤判定的解锁进入秒级短 burst。
- 定位请求不应设置过大的最小位移，否则进范围后可能迟迟没有新定位。
- 过期定位不能触发进入范围、打开钉钉或下班范围确认；只能作为日志和刷新定位的参考。
- 不能只依赖解锁广播。能收到 `USER_PRESENT/SCREEN_ON` 时必须记录日志并强制短 burst；收不到时，前台服务 tick 也要轮询解锁状态，检测到已解锁后强制短 burst 刷新定位，不能因为旧位置仍在范围外就跳过探测。
- 上班失败必须落成“上班未确认”状态，不能阻塞下班；下班按正常下班时间继续执行。
- 用户手动打开钉钉后，应用仍要发出可确认通知。
- “钉钉进入前台后自动记录”不能把 `startActivity()` 成功当作钉钉已打开；必须通过使用情况访问查询到钉钉前台事件，否则只提醒手动确认。
- 自动拉起钉钉不能只发一次。若前台验证失败，并且仍处于打卡窗口且手机已解锁，必须进入短重试窗口。
- 后续排查必须能从日志看到：窗口、定位、距离、解锁状态、是否尝试打开钉钉。
